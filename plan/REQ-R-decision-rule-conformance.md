## Background

The companion (§3.4) mandates the integer cutoff as the binding decision artefact for a baseline-derived (regression/empirical) test: `p* = WilsonLower(p̂_baseline, n_test, α)`, `c = ⌈n_test · p*⌉`, **`PASS iff K ≥ c`** where `K` is the raw observed success count. The compliance procedure (§3.2/§3.6, threshold *given*) is different: there the test sample's own Wilson lower bound must clear the threshold. baseltest was found applying the compliance shape on the empirical path — a procedure conflation, substantially over-conservative — and no fixture caught it, for two reasons that both concern feotest:

1. `threshold_derivation.json` has published `cutoff_integer` and `achieved_size` — the binding decision artefacts — for several releases, and **no framework in the family consumes them**. The oracle published the truth; nothing forced anyone to read it.
2. There was no scenario-level suite exercising the composed decision. A framework can be arithmetically conformant on every component and still compose them into the wrong rule.

feotest's status on the rule itself is **unknown** (it does not reference the cutoff fields anywhere). The suite below establishes it either way; per the owner ruling, red-then-green — land the conformance test first, so feotest's true status is demonstrated, not asserted. If it comes back red, the fix lands separately with the consequence stated plainly.

## Fixture shape

Suite `regression_decision`, tolerance `1e-10`. Each case carries a `procedure` discriminator:

```text
procedure: REGRESSION                     # companion §3.4 — threshold derived from baseline
  inputs:   { baseline_successes, baseline_trials, test_samples, confidence, observed_successes }
  expected: { threshold_real, wilson_lower, cutoff_integer, displayed_rate, achieved_size, verdict }

procedure: COMPLIANCE                     # companion §3.2/§3.6 — threshold given
  inputs:   { threshold, test_samples, confidence, observed_successes }
  expected: { wilson_lower, verdict }
```

Thirteen cases: the §3.4 worked example (p̂ = 0.951, n_test = 100 → p* ≈ 0.902124, c = 91, achieved size ≈ 0.024986) with `K = c` / `K = c − 1` boundaries; small-n discretisation cases where the real-valued bound and the integer cutoff disagree materially; the perfect-baseline `k = n` edge (§4.3.2 effective-rate guard); a compliance sibling group; and a **conflation-detector pair** sharing one observation across both procedures with opposite verdicts — a framework applying the compliance rule on the regression path fails the suite even though every component computation is arithmetically conformant.

For `REGRESSION` cases the verdict is binding via `K ≥ cutoff_integer`; `threshold_real`, `displayed_rate`, and `achieved_size` are the report obligations §3.4 places on a conformant verdict record.

New alongside the suites: **`manifest.json`** (`manifestVersion` 1), generated by the oracle — per-suite case rosters, binding vs informational expected-field classification, per-suite MD5 content hashes, and the family-mandatory tier: `wilson_ci`, `wilson_lower`, `threshold_derivation`, `verdict`, `latency_percentile`, `latency_threshold`, `regression_decision`.
