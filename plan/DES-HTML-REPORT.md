# DES-HTML-REPORT — HTML report for feotest

## Feature reference

Inventory code: **RP04** — HTML report (standalone, embedded CSS)

Specification:
[`inventory/catalog/reporting/RP04-html-report/README.md`](../../inventory/catalog/reporting/RP04-html-report/README.md)

## Goal

Generate a self-contained HTML file from a slice of `VerdictRecord`s that a
developer can open in any browser, attach to a pull request, or archive with
build artefacts. The report is functionally comparable to punit's HTML report
while remaining idiomatic Rust throughout.

## Approach: write raw HTML with `std::fmt::Write`

The report is a single function that writes HTML strings into a `String` via
`std::fmt::Write`. No template engine, no external crate.

**Rationale:**

- punit's report uses `StringBuilder` concatenation. The Rust equivalent is
  `std::fmt::Write` into a `String`.
- The HTML is structural and repetitive — no conditionals complex enough to
  justify a template DSL.
- Adding a template engine (Askama, Tera, Maud) would be the idiomatic choice
  for a web application, but this is a single standalone file with embedded CSS,
  not a web app. The report structure is fixed and well-defined.
- Zero additional dependencies. The framework already uses `serde`, `serde_yaml`,
  and `sha2`; adding a template crate for one output file is unjustified.
- An `HtmlEscaper` helper (see below) keeps the escaping correct without
  external help.

## Public API

```rust
// src/reporting/html.rs

/// Generates a standalone HTML report from verdict records.
pub struct HtmlReportWriter;

impl HtmlReportWriter {
    /// Generates the complete HTML content for a suite report.
    ///
    /// The returned `String` is a valid HTML5 document with all CSS
    /// embedded in a `<style>` block. It can be written directly to a
    /// file.
    #[must_use]
    pub fn generate(records: &[VerdictRecord]) -> String;
}
```

The caller is responsible for writing the returned string to disk. This
keeps the report generator pure (no I/O) and testable.

Re-export from `src/reporting/mod.rs`:

```rust
mod html;
pub use html::HtmlReportWriter;
```

## Document structure

The HTML follows the RP04 specification closely, matching punit's layout:

```
<!DOCTYPE html>
<html lang="en">
<head>
    <meta charset="UTF-8">
    <meta name="viewport" content="width=device-width, initial-scale=1.0">
    <title>feotest Report</title>
    <style>/* all CSS embedded here */</style>
</head>
<body>
    <header>
        <h1>feotest Report</h1>
        <p class="timestamp">Generated: {timestamp}</p>
        <div class="summary-stats">
            Total: N | Pass: N | Fail: N | Inconclusive: N
        </div>
        <!-- Inconclusive banner when applicable -->
    </header>

    <details class="assumptions">
        <!-- Statistical assumptions and limitations -->
    </details>

    <main>
        <section class="test-group">  <!-- one per use-case-id -->
            <h2>{use_case_id}</h2>
            <table>
                <thead>
                    Test Name | Verdict | Functional | p50 | p95 | p99 | Samples | Elapsed
                </thead>
                <tbody>
                    <!-- one row per verdict, with expandable detail -->
                </tbody>
            </table>
        </section>
    </main>

    <footer>
        Generated by feotest at {timestamp}
    </footer>
</body>
</html>
```

### Differences from punit

| Aspect | punit | feotest |
|--------|-------|---------|
| Title | "PUnit Test Report" | "feotest Report" |
| JUnit column | Separate JUnit/PUnit verdict columns | Single "Verdict" column (no JUnit layer) |
| Inconclusive banner | References `./gradlew exp` | References `cargo test` with experiment re-run guidance |
| Statistical analysis detail | Rendered via `VerdictTextRenderer` | Rendered directly from `StatisticalAnalysis` fields |
| Latency cells | p50/p95/p99 fixed columns | p50/p95/p99 fixed columns (same layout) |
| Timestamp | Java `DateTimeFormatter` | `chrono::Local::now()` or manual UTC — see dependency note |

## Content sections per verdict row

Each verdict row is expandable via `<details>/<summary>`:

### Level 1: summary table row

| Column | Source |
|--------|--------|
| Test Name | `identity.test_name()` falling back to `identity.use_case_id()` |
| Verdict | `verdict()` — colour-coded via CSS class |
| Functional | `functional.successes()`/total |
| p50, p95, p99 | From `latency().observed_percentiles()` — colour-coded against thresholds |
| Samples | `execution.samples_executed()`/`execution.samples_planned()` |
| Elapsed | `execution.cost().total_time()` |

### Level 2: verdict summary (expanded)

A `<pre>` block showing:

- Verdict and reason: `PASS — 0.9500 >= 0.9000`
- Intent: `VERIFICATION` / `SMOKE`
- Functional: `95/100 (0.9500)` — pass rate with threshold
- Latency evaluations (when present): per-percentile observed vs threshold
  with status
- Covariate misalignment guidance (when applicable): per-key
  baseline vs observed values

### Level 3: statistical analysis (nested `<details>`)

- Confidence level, CI bounds `[lower, upper]`
- Standard error
- Threshold and origin (SLA/SLO/Policy/Empirical)
- z-statistic and p-value (when present)
- Spec provenance: filename, contract reference
- Baseline provenance: source file, date, sample count, baseline rate,
  derived threshold
- Warnings

## Grouping

Verdicts are grouped by `identity.use_case_id()`. Each group becomes a
`<section class="test-group">` with its own `<h2>` heading and table. The
grouping preserves insertion order (use `IndexMap` from the `indexmap` crate,
or collect into a `Vec` of `(key, Vec<VerdictRecord>)` to avoid a new
dependency).

**Decision: no new dependency.** Group by iterating into a
`Vec<(String, Vec<&VerdictRecord>)>` preserving first-seen order. This avoids
adding `indexmap` for a single use site.

## CSS / visual design

All CSS is embedded in a `<style>` block. The design matches punit's visual
language:

```
:root {
    --pass-color: #2e7d32;
    --fail-color: #c62828;
    --inconclusive-color: #6a1b9a;
    --border-color: #dee2e6;
    --bg-light: #f8f9fa;
    --bg-white: #ffffff;
    --text-color: #212529;
    --text-muted: #6c757d;
}
```

| Element | Style |
|---------|-------|
| Pass verdict | Green text (`--pass-color`) |
| Fail verdict | Red text (`--fail-color`) |
| Inconclusive verdict | Purple text (`--inconclusive-color`) |
| Latency pass | Green text |
| Latency fail (strict) | Red text |
| Latency advisory warn | Amber text (`#f9a825`) |
| Warning callout | Yellow-left-border box |
| Misalignment guidance | Purple-left-border box |
| Statistical assumptions | Collapsible `<details>`, muted header |

Font stack: `-apple-system, BlinkMacSystemFont, 'Segoe UI', Roboto, sans-serif`
— system fonts, no external font loading.

Detail levels (`pre.level2`, `pre.level3`) use indented, bordered blocks with
monospace rendering for statistical data.

Tooltips on statistical terms use CSS-only `data-tip` hover via `::after`
pseudo-element — no JavaScript.

## Self-containment invariants

- No `<link>` to external stylesheets
- No `<script>` tags
- No external image references
- Collapsible sections via `<details>/<summary>` (native HTML5)
- Valid HTML5

## HTML escaping

A small internal helper escapes `&`, `<`, `>`, `"` in dynamic content:

```rust
fn escape(text: &str) -> String {
    text.replace('&', "&amp;")
        .replace('<', "&lt;")
        .replace('>', "&gt;")
        .replace('"', "&quot;")
}
```

All user-supplied text (test names, warning messages, covariate values, file
paths) passes through `escape()` before being written into the HTML.

## Timestamp

The report needs a wall-clock timestamp. Options:

1. **`chrono` crate** — the standard Rust choice for date/time formatting.
   Widely used, well-maintained.
2. **`time` crate** — lighter alternative.
3. **Accept a pre-formatted string** — push formatting to the caller.

**Decision: accept an `Option<&str>` timestamp parameter.** This keeps the
report writer free of date/time dependencies and makes it fully deterministic
for testing. The caller passes `chrono::Local::now().format(...)` or whatever
suits their context. When `None`, the report omits the timestamp line.

Updated signature:

```rust
pub fn generate(records: &[VerdictRecord], timestamp: Option<&str>) -> String;
```

## Statistical assumptions section

Matches punit's collapsible section verbatim — the statistical assumptions are
framework-agnostic. The text is embedded as a string constant.

Internal structure of `html.rs`:

- `pub struct HtmlReportWriter` — the public entry point
- `fn write_css(w: &mut String)` — embeds the `<style>` block
- `fn write_header(w: &mut String, records: &[VerdictRecord], timestamp: Option<&str>)` — header + summary stats
- `fn write_assumptions(w: &mut String)` — statistical assumptions section
- `fn write_group(w: &mut String, use_case_id: &str, records: &[&VerdictRecord])` — one test group
- `fn write_verdict_row(w: &mut String, record: &VerdictRecord)` — one table row with expandable detail
- `fn write_verdict_detail(w: &mut String, record: &VerdictRecord)` — level 2 summary
- `fn write_statistical_detail(w: &mut String, record: &VerdictRecord)` — level 3 analysis
- `fn write_latency_cell(w: &mut String, ...)` — single latency cell with colour
- `fn write_misalignment_guidance(w: &mut String, record: &VerdictRecord)` — covariate misalignment box
- `fn write_footer(w: &mut String, timestamp: Option<&str>)`
- `fn escape(text: &str) -> String` — HTML escaping
- `fn group_by_use_case(records: &[VerdictRecord]) -> Vec<(String, Vec<&VerdictRecord>)>`
- `fn verdict_css_class(verdict: Verdict) -> &'static str`

## Testing strategy

### Snapshot tests (`insta`)

The primary testing mechanism. Build `VerdictRecord`s covering key scenarios
and snapshot the full HTML output. This catches regressions in structure,
content, and CSS.

Snapshots:

| Snapshot | Scenario |
|----------|----------|
| `html_single_pass` | One passing verdict, no latency |
| `html_single_fail` | One failing verdict with full stats |
| `html_inconclusive_covariate` | Inconclusive due to misalignment |
| `html_with_latency` | Verdict with latency evaluations (pass + advisory warn) |
| `html_latency_strict_fail` | Latency strict violation |
| `html_multi_group` | Multiple use cases, multiple verdicts |
| `html_warnings` | Verdict with warnings |
| `html_empty` | Empty record slice |

### Unit tests

- `escape()` handles all special characters
- `group_by_use_case()` preserves order, groups correctly
- `verdict_css_class()` returns correct classes
- Summary counts match record contents

### Integration test

A `tests/html_report.rs` integration test that generates a report from
realistic verdict records and asserts:

- Output starts with `<!DOCTYPE html>`
- Contains `<style>` block
- Contains no `<link>` or `<script>` tags
- Contains expected test names
- Contains expected verdict text

## Dependencies

**No new dependencies required.**

- HTML generation: `std::fmt::Write` + string concatenation
- HTML escaping: hand-written (4 replacements)
- Grouping: `Vec<(String, Vec<&VerdictRecord>)>`
- Timestamp: caller-provided string
